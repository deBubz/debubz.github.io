# Relational v Non-relational

Having past experience with Relational Databases, I want to better understand the Non-Relational Database, MongoDb my team is using to store sensor data. Here are the findings

RelationalDB Non-RelationalDB (MongoDb)
Fixed format, requires to define a schema, better data integrity. More Flexible structure, better for data that changes frequently
Data relation, better at seperating and linking related data For, MongoDb, data is recorded as seperated,unrelated doccuments

Due to the limitation of the PubSubClient`s payload publication(128 bytes per payload). My team had aggreed upon a sending a payload in the format of a string consist of 12 characters(generally alphanumerical chars are 1 byte yeach). The server will then set up to splice up the string into seperate categories and stored as a doccument in the DB.

# Day 4 Improving the PoC

With Most of the basic functionalities done, I spent the day thinking about how I could further improve my solution's Proof of Concept

# Large Scale Design

Thinking about if my device would have been implimented in a large city using this doc as a reference, these are issues which needs to be address.

  • A difficult issue would by how data is transmitted. The current design is built with assumption that a Wifi connection is availible everywhere, at all times.
  • Power is another realistic issue. There are 2 ways of approaching this:
    • One is to modify current light poles to house the device and use the pole's power, but that would be rather expensive to replace everything.
    • Another method is to build the device as an attatchment to a light pole, powered by its own power supply.
  • With how the device functions, notifies the responsible organizations for street lights when they are non-functional and keep data on the bulb's life span. It is not necessary to continuously record the data. This should further help increase the lifespan if the device is built with its own power supply.
